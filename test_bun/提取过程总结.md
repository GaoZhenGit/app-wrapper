## 从claude.exe提取JS的完整过程

### 第一步：PE结构分析

**工具**：DIE (Detect It Easy) 和 hexdump

**发现**：
1. `claude.exe`是64位PE32+格式可执行文件
2. 由Bun打包生成（DIE识别为"Bun Pack"）
3. 包含12个PE节区，关键节区：`.bun`（124MB）

**关键命令**：
```bash
# 检查PE节区表
hexdump -C claude.exe | grep -A1 ".bun"
```

**PE节区信息**：
- `.bun` section: VirtualAddress=0x74A7000, SizeOfRawData=124842706字节
- 文件偏移: PointerToRawData=0x74A7800

---

### 第二步：提取.bun节区数据

**方法**：从PE文件中按文件偏移提取原始数据

```bash
dd if=claude.exe of=bun_section.bin bs=4096 skip=0x74A7800/4096 count=124842706/4096+1
```

**提取结果**：
- `bun_section.bin` (124.8MB) - 包含JS源码 + Bun bytecode的混合数据

**关键发现**：
- Bun打包格式 = JS源码（前段） + Bytecode（后段）
- JS源码以IIFE格式开头：`(function(exports, require, module, __filename, __dirname) {`
- Bytecode以二进制数据形式存储在JS代码之后

---

### 第三步：定位JS代码边界

**挑战**：bun_section.bin混合了JS和bytecode，需要精确分离

**定位方法1 - 查找JS结束标记**：
```bash
# 查找最后一个IIFE结束符号
python -c "
with open('bun_section.bin', 'rb') as f:
    data = f.read()
    pos = data.find('E_9();'.encode())
    print(f'E_9(); at offset: {pos}')
"
```

**定位方法2 - 查找bytecode起始**：
```bash
# 查找bytecode魔数特征
hexdump -C bun_section.bin | sed -n '122817400/512p'
```

**发现的边界**：
- **JS起始位置**: offset 451字节（跳过bun header元数据）
- **JS结束位置**: offset 13117810字节（`E_9();})` 之后）
- **精确大小**: 13117359字节（约13MB纯JS代码）

---

### 第四步：提取纯JS代码

**精确提取命令**：
```bash
dd if=bun_section.bin of=claude_pure.js bs=4096 skip=451 iflag=skip_bytes count=13117359 iflag=count_bytes
```

**提取结果**：
- `claude_pure.js` (13MB) - 包含完整JS源码但带CommonJS wrapper

**验证提取完整性**：
```bash
head claude_pure.js   # 显示: (function(exports, require, module, ...) {
tail claude_pure.js   # 显示: E_9();})
bun claude_pure.js --version  # 成功输出: 2.1.116 (Claude Code)
```

---

### 第五步：移除CommonJS wrapper

**问题分析**：
- Bun打包时添加CommonJS wrapper: `(function(exports, require, module, __filename, __dirname) { ... })()`
- 直接运行时wrapper内的`require`无法访问bun的模块系统
- 错误：`TypeError: require is not a function`

**移除wrapper的方法**：
```python
# Python脚本移除wrapper
with open('claude_pure.js', 'rb') as f:
    data = f.read()

wrapper_start = b'(function(exports, require, module, __filename, __dirname) {'
wrapper_end = b'})()'

# 去除开头wrapper
data = data[len(wrapper_start):]

# 去除结尾wrapper
data = data[:-len(wrapper_end)]

with open('claude_unwrapped.js', 'wb') as f:
    f.write(data)
```

**移除后结果**：
- `claude_unwrapped.js` (13MB) - 纯JS代码，无wrapper
- 文件开头: `// Claude Code is a Beta product...`
- 文件结尾: `...E_9();`

---

### 第六步：创建minimal loader

**问题**：eval执行代码时，bun不提供全局`require`

**解决方案** - minimal_loader.js：
```javascript
// 提供全局require
global.require = require;

// 执行unwrapped代码
const code = require('fs').readFileSync(__dirname + '/claude_unwrapped.js', 'utf-8');
eval(code);
```

**工作原理**：
1. 使用`global.require`将bun的require注入全局作用域
2. eval执行时可以访问全局变量
3. Claude代码中的`require("crypto")`等调用正常工作

## 关键技术总结

### 1. Bun打包格式结构

```
Bun可执行文件结构：
├── PE Header (标准PE头)
├── Code Sections (.text, .data等)
├── .bun Section (124MB)
│   ├── Bun Header (前451字节)
│   │   ├── 元数据标记: "// @bun @bytecode @bun-cjs"
│   │   ├── 入口文件路径: "file:///B:/~EBUN/root/src/entrypoints/cli.js"
│   │   └── 其他配置信息
│   ├── JS Source Code (13MB)
│   │   ├── CommonJS wrapper: (function(...) {
│   │   ├── 应用完整代码
│   │   └── 入口函数调用: E_9();})()
│   └── Bun Bytecode (109MB)
│       ├── 编译后的字节码数据
│       └── 运行时优化缓存
└── Resources / Import Tables
```

### 2. 分离JS和Bytecode的关键

**识别特征**：
- **JS代码特征**: ASCII文本，以`(function`开头，以`})()`结束
- **Bytecode特征**: 二进制数据，包含大量`0xFF`填充和非ASCII字节

**定位技巧**：
- 查找IIFE结束标记：`E_9();` 或 `})()`
- 使用strings命令找最后的有效JS代码位置
- hexdump查看边界处的数据特征

### 3. CommonJS wrapper的处理

**wrapper的作用**：
```javascript
(function(exports, require, module, __filename, __dirname) {
  // 应用代码在这里
})
```

**为何需要移除**：
- Bun runtime正常会调用这个wrapper并传入参数
- 直接eval时wrapper未被调用，参数都是undefined
- 移除wrapper让代码直接在全局作用域执行

**替代方案** - minimal loader：
- 通过`global.require`提供全局require
- eval执行代码可以访问全局变量
- 避免了复杂的wrapper调用问题

## 经验教训

### 失败尝试1 - 直接用bun运行bundle

```bash
bun claude_bundle.js -c  # 错误: require is not defined
```

**原因**: Bun未识别CommonJS wrapper，未正确初始化模块环境

### 失败尝试2 - bun build重新打包

```bash
bun build claude_bundle.js --compile --bytecode
```

**结果**: 生成的exe仍然报`require is not defined`错误

**原因**: 重新打包无法恢复原始bun runtime的初始化逻辑

### 失败尝试3 - 手动调用wrapper

```javascript
const bundleFunc = eval(bundleCode);
bundleFunc(exports, require, module, __filename, __dirname);
```

**错误**: `bundleFunction is not a function`

**原因**: IIFE已在定义时自动执行，eval返回undefined

### 成功关键

**核心突破点**：
1. **理解Bun打包格式**：JS + Bytecode混合存储
2. **精确分离技术**：找到13MB JS代码的准确边界
3. **移除wrapper**：让代码直接在eval作用域执行
4. **minimal loader**：只提供缺失的全局require
